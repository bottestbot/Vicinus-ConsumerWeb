### Analytics endpoint ‚Äî reporting integrity (P0)
_**Behaviour today:** `POST /analytics/listing-event` is public, unauthenticated and unthrottled, and every field is caller-supplied. Anyone on the internet can assert that an event happened on vicinus.ca, and CREA receives it stamped with our `DestinationID`. The service docstring claims client throttling is unnecessary because "CREA dedups by UUID within 5-minute windows" ‚Äî but the **caller chooses the UUID**, so rotating it defeats the dedup entirely. **Target behaviour:** Vicinus reports to CREA only what Vicinus actually observed ‚Äî the endpoint should be an internal sensor, not a public megaphone. **Note:** CORS (`main.ts:17`) does not mitigate this; it is a browser policy and is ignored by `curl`/scripts._

- [ ] **CREA-01** [P0] _(Security)_ **Rate-limit the analytics endpoint + trust the proxy** ‚Äî `@nestjs/throttler` is **not installed** and the setup is a commented-out TODO. Install it, wire `ThrottlerModule.forRoot` in `AppModule` + `useGlobalGuards` in `main.ts`, and put a tight per-route `@Throttle` on `listing-event` (~30/min/IP is generous for a human). Also set `app.set('trust proxy', 1)` ‚Äî without it `x-forwarded-for` is attacker-controlled (`analytics.controller.ts:24`) and per-IP limiting is trivially evaded. Highest value per minute spent; do this first. `S` ¬∑ _api/src/main.ts:10-11,25, api/src/app.module.ts, api/src/modules/analytics/analytics.controller.ts_
- [ ] **CREA-02** [P0] _(Security)_ **Issue the visitor UUID server-side instead of accepting it from the body** ‚Äî root-cause fix. Today `getDeviceId()` mints a `crypto.randomUUID()` into `localStorage` and the browser sends it up, so the identity is fully caller-chosen and CREA's 5-minute dedup can never bind. Instead issue the identifier on first visit as an `HttpOnly` signed cookie, read it from the cookie in the controller, and **ignore `dto.uuid`** entirely (drop the field, or keep it and discard). Restores dedup and makes the visitor identity non-forgeable. `M` ¬∑ _api/src/modules/analytics/analytics.controller.ts, dto/listing-event.dto.ts, src/lib/deviceId.ts_
- [ ] **CREA-03** [P1] _(Security)_ **Validate `listingKey` before forwarding to CREA** ‚Äî `listingKey` is accepted as any string ‚â§64 chars, so events can be reported for listings we don't serve or that don't exist. One cache lookup; kills fabricated-listing spam. `S` ¬∑ _api/src/modules/analytics/crea-analytics.service.ts_
- [ ] **CREA-04** [P2] _(Improvement)_ **Queue CREA LogEvents calls instead of 1:1 fan-out** ‚Äî each inbound request currently triggers an immediate outbound GET to `analytics.crea.ca`, so a burst amplifies straight through to CREA under our DestinationId. ### Reconciliation ‚Äî destructive cron safety
_**Behaviour today:** the 15-min property sync is upsert-only and never removes, so sold/cancelled/expired listings accumulate forever ‚Äî which DDF terms prohibit. The new `@Cron(EVERY_DAY_AT_4AM) reconcileListings()` fixes that by deleting any local `Property` absent from CREA's `PropertyReplication()` master list. It guards against an **empty** master list, but the list arrives **paginated** via `@odata.nextLink` ‚Äî if the chain breaks partway and the response still parses, the job holds a short-but-nonzero set and deletes everything missing. The system currently cannot distinguish "most listings sold overnight" from "we only received part of the answer", and treats both as an instruction to delete. **It has never been run; its first execution is against prod, unattended, at 4AM.** **Target behaviour:** delete only when confident the master list is complete ‚Äî a partial answer must delete nothing._
_**Risk framing:** *not* reconciling is the actual contract violation (advertising sold inventory); reconciling *wrongly* is an operational failure, not a compliance one. **Blast radius is contained** ‚Äî `SavedProperty`/`VisitedProperty.propertyId` are plain DDF ListingKey strings (not FKs) so user saves survive, and `Alert.propertyId` is `onDelete: SetNull` so alerts are orphaned but not lost. Recovery is a re-sync, not a restore. What degrades meanwhile: neighbourhood inventory counts + listings, valuation comparables, dashboard cards, alert matching (verified consumers of the local `Property` table: `properties.service.ts`, `neighbourhoods.service.ts`, `users.service.ts`, `alerts.service.ts`)._
